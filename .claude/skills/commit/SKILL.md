# Commit Messages

Follow these conventions when creating commits.

## Prerequisites

Before committing, ensure you're working on a feature branch, not the main branch.

```bash
git branch --show-current
```

If you're on `main` or `master`, create a new branch first:

```bash
git checkout -b <type>/<short-description>
```

Branch naming: `<type>/<short-description>` (e.g., `feat/add-overtone-construct`, `fix/schema-validation`).

## Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

The header is required. Scope is optional. All lines must stay under 100 characters.

### Package Scopes

In this monorepo, scope commits to the affected package:

| Scope | Package |
|-------|---------|
| `amplify-overtone` | `packages/amplify-overtone/` — backend |
| `amplify-overtone-client` | `packages/amplify-overtone-client/` — frontend |
| `integration-tests` | `packages/integration-tests/` — E2E tests |
| `test-infra` | `packages/test-infra/` — CDK test infrastructure |

Examples: `feat(amplify-overtone): Add auth propagation Lambda`, `fix(amplify-overtone-client): Handle missing config`.

For changes spanning multiple packages or affecting the root (CI, workspace config), omit the scope or use a general scope like `deps` or `ci`.

## Commit Types

| Type      | Purpose                           |
| --------- | --------------------------------- |
| `feat`    | New feature                       |
| `fix`     | Bug fix                           |
| `ref`     | Refactoring (no behavior change)  |
| `perf`    | Performance improvement           |
| `docs`    | Documentation only                |
| `test`    | Test additions or corrections     |
| `build`   | Build system or dependencies      |
| `ci`      | CI configuration                  |
| `chore`   | Maintenance tasks                 |
| `style`   | Code formatting (no logic change) |
| `meta`    | Repository metadata               |
| `license` | License changes                   |

## Subject Line Rules

- Use imperative, present tense: "Add feature" not "Added feature"
- Capitalize the first letter
- No period at the end
- Maximum 70 characters

## Body Guidelines

- Explain **what** and **why**, not how
- Use imperative mood and present tense
- Include motivation for the change

## Footer: Issue References

```
Fixes GH-1234
Fixes #1234
Refs LINEAR-ABC-123
```

## AI-Generated Changes

When changes were primarily generated by a coding agent, include the Co-Authored-By attribution:

```
Co-Authored-By: Claude <noreply@anthropic.com>
```

Do not add phrases like "Generated by AI" or "Written with Claude" anywhere else.

## Examples

### Simple fix

```
fix(amplify-overtone): Handle missing configuration ARN in IAM policy

The handler Lambda was getting AccessDenied when a configuration was
attached to the resource. Add configuration ARN to the PolicyStatement
resources list.

Fixes #12
Co-Authored-By: Claude <noreply@anthropic.com>
```

### Feature with scope

```
feat(amplify-overtone): Add auth propagation Lambda

Wire Cognito user pool into the overtone construct so auth tokens are
automatically validated before handler invocation.
```

### Breaking change

```
feat(types)!: Rename OvertoneProps.config to overtoneConfig

Use explicit overtoneConfig to clarify the configuration object
structure and avoid ambiguity with other config fields.

BREAKING CHANGE: OvertoneProps.config is now overtoneConfig
```

## Changesets

This project uses `@changesets/cli` to track version bumps. When your commit affects the published package (new feature, bug fix, breaking change), you need a changeset:

```bash
pnpm changeset
```

This creates a `.changeset/<random-name>.md` file. Commit it alongside your code:

```bash
git add .changeset/ packages/amplify-overtone/src/
git commit -m "feat(amplify-overtone): Add auth propagation Lambda"
```

**When to create a changeset:**
- `feat` — always (minor bump)
- `fix` — always (patch bump)
- `ref` / `perf` — if it changes published output (patch bump)
- `docs` / `test` / `build` / `ci` / `chore` / `style` — skip (no changeset needed)

**Rule:** If it doesn't affect what consumers get from `npm install`, skip the changeset.

## Principles

- Each commit should be a single, stable change
- Commits should be independently reviewable
- The repository should be in a working state after each commit
